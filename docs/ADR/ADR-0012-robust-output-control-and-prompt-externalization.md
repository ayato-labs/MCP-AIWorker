# ADR-0012: サブLLMのための堅牢な出力制御とプロンプトの外部化

- **日付**: 2026年6月20日
- **ステータス**: 承認済み
- **決定者**: ayato-labs、Geminiエージェント

## 背景
`draft_code`ツールは、コード生成の負荷の高い処理を低コストのサブLLMに依存しています。これらのモデルは、会話的なフィラー、前文/後文といった「ノイズ」を生成したり、コンテキスト/トークン制限のために出力が切り詰められたりすることがあります。

既存の実装では、Markdownブロックに対して単純な文字列置換を使用していましたが、これは脆弱であり、LLMの出力が不正な形式であったり、予期しないテキストが含まれていたりすると、コードベースが破損する可能性がありました。


## 決定事項
堅牢なコード抽出とシステム安定性の維持のため、以下の対策を実施します。

1. **プロンプトの外部化**:

- `server.py` にハードコードされたシステムプロンプトを外部ディレクトリ (`prompts/`) に移動します。

- これにより、コアロジックを変更することなく、モデルごとのチューニングが容易になります。

2. **XMLベースの出力構造化**:

- システムプロンプトに、すべてのコード出力が `<draft_output>` および `</draft_output>` タグで囲まれるという厳格な制約を適用します。

3. **多段階解析ロジック**:

- `clean_code_output` 内の単純なクリーニング処理を、堅牢な解析パイプラインに置き換えます。

- **プライマリ**: 一致するXMLタグ内のコンテンツを抽出します。

- **レスキュー**: 終了タグがない場合 (切り捨てを示す場合)、開始タグ以降のすべてのコンテンツをレスキューします。

- **フォールバック**: XMLタグが存在しない場合は、Markdownコードブロックの抽出にフォールバックします。

- **ノイズ除去**: 一般的なLLM会話フレーズを除去する最終処理です。

## 影響
### メリット
- **破損リスクの低減**: 会話中の「ノイズ」がソースファイルに書き込まれるのを防ぎます。

- **安定性**: 「レスキュー」ロジックにより、出力が切り詰められた際の障害や無限リトライループを防ぎます。

- **保守性**: システムプロンプトは、サーバーコードとは独立して更新できます。

### デメリット／リスク
- **トークンオーバーヘッド**: XMLタグを明示的に要求することで、サブLLMの入力トークン数がわずかに増加します。

- **プロンプト準拠**: 低コストのモデルでは、XMLラッパーが無視される場合がありますが、Markdownフォールバックによりこのリスクは軽減されます。
